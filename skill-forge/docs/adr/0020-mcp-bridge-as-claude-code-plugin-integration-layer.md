# ADR-0020: MCP Bridge as the Claude Code Plugin Integration Layer

**Date:** 2026-04-12
**Status:** Proposed
**Deciders:** skill-forge maintainers
**Supersedes:** N/A

## Context and Problem Statement

The context-bazaar plugin needs to integrate with Claude Code so users can
discover and use knowledge artifacts without leaving their editor. The
compiled `dist/claude-code/` artifacts are already CLAUDE.md files that
Claude Code reads as context — but passive file distribution is the floor,
not the ceiling. Users need to be able to ask the assistant *which*
artifacts exist, filter by collection, and read artifact content
interactively. The plugin packaging format
(`.claude-plugin/`, `.mcp.json`) supports both passive skill files and
active MCP servers; the question is which integration layer to choose.

## Decision Drivers

- Users should be able to discover available artifacts conversationally
  from inside Claude Code, not by browsing a website
- The plugin must work with a single plugin install — no additional setup
  commands on the user's machine
- The bridge must be distributable as a committed file (no post-install
  build step required)
- Adding new catalog tools later should not require changes to the
  plugin manifest

## Considered Options

1. **Pure skill file distribution** — ship only the compiled
   `dist/claude-code/*.md` files as the plugin's `skills/` directory.
   No MCP server.
2. **Custom slash command** — ship a `.claude/commands/` script that
   reads `catalog.json` and prints a summary
3. **MCP server exposing catalog tools** — a bundled `mcp-server.cjs`
   that implements the MCP tool protocol and reads `catalog.json`

## Decision Outcome

**Chosen option: Option 3 — MCP server**, because it is the only option
that supports interactive, parameterised catalog queries (filter by
collection, type, or name) without requiring Claude Code to have the full
catalog pre-injected into every context window.

The bridge is pre-bundled into a single `bridge/mcp-server.cjs` using
`bun build --format cjs`. This removes the need for users to run
`npm install` inside the plugin directory — the bridge is self-contained
and ready to execute with `node` alone. The bundle includes the
`@modelcontextprotocol/sdk` dependency inline.

Three tools are exposed:
- `catalog_list(collection?, type?)` — returns filtered catalog entries
- `artifact_content(name)` — reads a specific artifact's `knowledge.md`
- `collection_list()` — returns collections with member counts

### Positive Consequences

- Users can ask *"what's in the neon-caravan collection?"* or *"show me
  the commit-craft artifact"* without leaving Claude Code
- The bridge reads from `catalog.json` which is regenerated by
  `forge publish` — no per-artifact wiring required for new content
- Adding new tools requires only a change to `src/mcp-bridge.ts` and a
  rebuild of the bundle; the `.mcp.json` manifest doesn't change
- `CLAUDE_PLUGIN_ROOT` environment variable is respected so the bridge
  works whether installed as a plugin or run locally for development

### Negative Consequences / Trade-offs

- The pre-bundled CJS (0.48 MB) is committed to the repository — it
  must be rebuilt and recommitted on every release via `forge publish`
- Users who want only the static skill files get the MCP server too,
  which starts a background process when the plugin loads
- The MCP bridge exposes read-only catalog access; `forge install` is
  intentionally not exposed to avoid unintended side effects from within
  the assistant

## Options Analysis

### Option 1: Pure skill file distribution
**Pros:** Zero runtime overhead; no background process; familiar CLAUDE.md
format
**Cons:** No interactivity; the full catalog must be injected as context
regardless of whether the user wants it; no filtering

### Option 2: Custom slash command
**Pros:** Simple; no MCP complexity; printable catalog summary
**Cons:** Slash commands are synchronous and output-only; cannot accept
parameters or return structured data for follow-up queries

### Option 3: MCP server (chosen)
**Pros:** Interactive, parameterised, composable; standard protocol;
extensible without manifest changes
**Cons:** Bundle size; committed binary; background process overhead
